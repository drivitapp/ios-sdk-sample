Drivit

Public interface for the client. This will be the main access to the Drivit SDK funcionalities.

  • shared

    Singleton instance of the framework

  • googleAPIKey

    Sets the google api key that Drivit will use e.g. to snap the trip locations to the nearest roads. This api key is safely secured on the sdk’s side using state-of-the-art encryption

  • plan

    Manages data plan

  • precision

    Manages location precision

  • mode

    Manages recording mode

  • screen

    Manages screen off time

  • isUserLogged

    Determines if Drivit is running.

    Returns all the charging rules configured in the SDK. If the case of a new account, it provides default rules. The order is not guaranteed

  • isWaitingForConsumptionData(guid:)

    Whether the SDK is still waiting for consumption data to arrive from Drivit servers or not

  • fuelConsumption(guid:)

    Returns the fuel consumption of the trip, in l/100km. This method may return one of three cases:

    1. If the user manually edited the consumption of the trip, it will return the value inserted
    2. If a good-accuracy fuel consumption estimate exists, it will return it
    3. If the consumption analysis wasn’t performed yet or if it’s low-accuracy estimate, it will fallback to the vehicle average as provided by{@link VehicleSettings#getCurrentConsumptionValue()}
  • fuelCost(guid:)

    The money spent in fuel in the trip as a function of the fuel consumption given by {@link #getFuelConsumption()}

Trips
  • hasAutomaticClassification(guid:)

    Method to determine whether the trip has been processed by the Drivit server and has its automatic classification available

    • guid: trip identifier
  • rejectedReason(guid:mergeWithServer:)

    Returns the rejected reason or, in other words, the validity state of this trip. The default value is -2, meaning no conclusion has been taken as to if the trip belongs to the user or not.

    • guid: trip identifier
    • mergeWithServer: false to retrieve a value without analysis from Drivit server (e.g. the user may reject the trip directly on the phone), true if you want to merge both results. When merging, any potential user input will always have precedence over the analysis from the Drivit server.
  • handleUserAction_userArrived()

    Method that stops an ongoing trip, typically because the user arrived at its location. This method is only valid if the SDK is recording a trip and should only be called if the user explicitly tells the app to end it, e.g. through a I’ve arrived! action button

  • isRecordingTrip()

Other methods
  • settings()

    Returns the status of the framework. This class will give you the status of the framework and some permissions needed by the framework.

  • runningTrip()

    Call this method to get the trip that is running at the moment.

  • delegate(_:)

    To setup a delegate to listen to events given by the SDK. This events are for example: Trip did start, Trip did stop

    • delegate: A delegate that conforms to the DrivitModelDelegate protocol will receive the events.
  • hasFilesToSinc(TripGUID:)

    This method will say if there is any files needed to upload with the metadata of this trip.

  • checkIfUserHasTripsOrAccelToSync()
  • forceSyncOfTrip(TripGUID:)

    Force the sync of the files that still need to be uploaded to the server, if the trip has no files to be synced this method will not do anything.

  • syncAccelAndTrips(includeScoringData:completion:)

    Force-syncs all trip-related info, even if the user is not on wifi. Take care with this method as it can consume a large amount of mobile data.

    • includeScoringData: argument that specifies which data should be synced. If false, it will only sync data required to estimate the trip consumptions. If true, it will also sync data required for scoring and trip classification.
